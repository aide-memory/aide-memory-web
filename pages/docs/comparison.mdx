# Comparison

aide-memory, [claude-mem](https://github.com/thedotmack/claude-mem), and [engram](https://github.com/ayvazyan10/engram) all attempt to give AI coding agents persistent memory across sessions. They take different shapes. This page describes each factually so you can pick the one (or combination) that matches your workflow.

## What's distinctive about aide-memory

A short list of traits that aide-memory carries together. Each individual trait isn't unique, but the combination is what aide-memory is built around:

- **Categorized + scoped recall**, four memory layers (preferences / technical / area_context / guidelines) plus glob scopes (e.g. `src/auth/**`, `packages/api/**`). The agent gets memories relevant to the area it's working in, ordered by how specific the layer is.
- **Hooks prompt at the right moments**, six editor hooks fire across the session lifecycle so capture and recall happen without the agent (or you) needing to remember to call a tool.
- **File-per-memory in your repo**, each memory is a JSON file under `.aide/memories/`. `git add`, `git push`, `git pull` is the team-sync path. Personal preferences are gitignored; team-shared memories travel with the repo.
- **Cross-tool memory store**, Claude Code and Cursor read the same `.aide/` directory today; further editor support may come based on user feedback.
- **Local-first**, SQLite cache and JSON files on your machine. Anonymized usage counts ship to PostHog by default; disable with `AIDE_TELEMETRY=off`.

The grid below maps how each of the three tools covers each axis.

| **Capture mechanism** | Six hooks pre-wired across the session lifecycle at `aide-memory init` | Session-end summarization that emits the next session's context | Opt-in session-end hook script you copy in and configure |
| **Recall mechanism** | Small (~20 token) nudge per file read; agent decides which memories to fetch | Inject context from recent sessions at session start; 3-layer MCP search workflow (search → timeline → get_observations) on demand | Namespace-scoped retrieval against typed memory store |

## How each tool's approach differs

**Per-file nudge vs session-start dump.** aide-memory injects a small (~20 token) nudge when the agent opens a file with scoped memories and lets the agent decide whether to call `aide_recall`. claude-mem injects context from recent sessions at session start and exposes search tools for on-demand detail. engram retrieves from typed namespaced stores. Different tradeoffs: per-file recall is precise but depends on file reads; session-start injection is comprehensive but pays a token cost up front.

**Glob-pattern path scoping inside a project.** A memory about test utilities can be scoped to `tests/**` so it surfaces only when the agent is in test code. claude-mem scopes by folder via Folder Context Files (per-folder CLAUDE.md activity timelines). engram scopes by namespace (per project or agent). Same goal, different granularities.

**Hooks pre-wired at init.** aide-memory configures hooks at six lifecycle events (SessionStart, PreToolUse, PostToolUse, UserPromptSubmit, Stop, PreCompact) the moment you run `aide-memory init`. engram offers an opt-in session-end hook script you wire in. claude-mem handles capture through its session lifecycle. If you want capture wired up without manual config, that's the aide-memory default.

**Files as storage substrate.** aide-memory's memories live as JSON files inside `.aide/` next to your code, so `git add` / `git commit` / `git pull` is the team-sync path. claude-mem and engram use vector / typed stores instead, which trade the file-level workflow for richer query semantics. Pick the model that matches how your team already works.

**Four typed layers vs three vs none.** aide-memory: preferences (how a contributor works), technical (facts about the stack), area_context (decisions for a code area), guidelines (team principles). engram: Episodic / Semantic / Procedural (closer to a cognitive-architecture framing). claude-mem: folder timelines + observations (no fixed layer typology). All valid; the right one depends on the kinds of memories you want to capture.

**Personal vs team split.** aide-memory's preferences folder splits into `personal/` (gitignored) and `shared/` (committed) so personal style stays personal but team conventions travel with the repo. claude-mem and engram don't model this distinction at the storage layer.

## When to pick each

- **aide-memory** when you want layered, scoped, auto-captured memory that travels with the repo so your teammates' agents pick up what yours learned. Solo developer or team; aide-memory leans into both. Improving session handoff (so context from one session can carry to the next without manual capture) is on the roadmap.
- **claude-mem** when you want session-summary-driven context-on-startup tied to Claude Code's folder structure, with a search-and-drill-down workflow.
- **engram** when you want a tool-agnostic cognitive-architecture store with three typed memory categories and namespace isolation.

## Where aide-memory sits relative to rules files and skills

aide-memory isn't built to replace `CLAUDE.md`, `.cursorrules`, skills, or any other static-context mechanism your editor already provides. Those are great at "always-on, always-injected" knowledge: project-wide style, security rules, the things every turn needs.

aide-memory fills in the gaps those static surfaces leave open:

- **Knowledge that should only surface in some areas of code**, not in every turn.
- **Knowledge that should be captured automatically when corrections, decisions, or non-obvious findings happen**, not manually pasted into a rules file later.
- **Knowledge that needs to stay in sync across teammates** without each person manually editing the same `CLAUDE.md` and merging conflicts.
- **Knowledge that needs to be categorized** (preferences vs technical vs area decisions vs team guidelines) so recall can prioritize correctly.

Run aide-memory alongside your existing rules files and skills. They cover the static, always-relevant rails; aide-memory covers the dynamic, scoped, growing-with-the-code knowledge.
