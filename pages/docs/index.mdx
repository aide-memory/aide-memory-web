# AIDE Memory

**Structured, persistent, path-scoped memory for AI coding agents.**

AI coding agents have a memory problem. Claude has `CLAUDE.md`, Cursor has `.cursorrules`, but these are static files with no structure, no path scoping, and no automatic capture. Every new session starts from scratch. Your agent doesn't remember what you taught it yesterday.

aide-memory fixes that. Two minutes to set up. Zero config. No Docker, no cloud, no API keys.

```bash
npx aide-memory init
```

---

## Why aide-memory?

- **Your agent forgets corrections.** You told it to use `dayjs` not `moment`, next session, it reaches for `moment` again.
- **Context files don't scale.** A flat `CLAUDE.md` with 200 lines of rules becomes noise. No way to scope rules to specific code areas.
- **No automatic capture.** You have to manually write down what you taught the agent. Nobody does this.
- **Editor lock-in.** What you teach in Claude Code doesn't carry to Cursor, and vice versa.

aide-memory solves all four: structured layers, path-scoped recall, hook-driven auto-capture, and cross-editor via MCP.

---

## How It Works

### 1. Hooks capture context automatically

Six hooks fire during normal development, no manual effort required:

| Hook | When | What it does |
|------|------|-------------|
| **SessionStart** | Session begins or resumes | Injects top-N preferences + guidelines + priority-always memories |
| **PreToolUse** | Before file reads / edits / search | Nudges agent: "N memories exist for this path" (~20 tokens) |
| **PostToolUse** | After aide_recall / aide_remember / aide_search | Records recalled IDs so re-reads don't re-block |
| **UserPromptSubmit** | User corrects agent | Detects corrections, stores them as scoped memories |
| **Stop** | Task completion | Prompts agent to reflect and store learnings |
| **PreCompact** | Before context loss | Extracts decisions before they're compacted away |

### 2. Memories are scoped to code paths

A memory about authentication surfaces when the agent reads `src/auth/`, not when it reads `src/checkout/`. Path inheritance means `src/**` memories surface everywhere under `src/`.

### 3. Four structured layers

| Layer | Purpose | Example |
|-------|---------|---------|
| **preferences** | How you like to work | "Use single quotes, 2-space indent" |
| **technical** | Facts about the stack | "Apollo needs useGraphQLGateway: true" |
| **area_context** | Decisions for specific code | "Checkout uses Stripe, not PayPal" |
| **guidelines** | Team principles | "API responses must use camelCase" |

### 4. File-per-memory, git-synced

Each memory is a human-readable JSON file in `.aide/memories/`. Commit them, push them, pull them. Your team's agents share the same context.

## What You Get

- **7 MCP tools**, aide_recall, aide_remember, aide_update, aide_forget, aide_search, aide_memories, aide_import
- **13 CLI commands**, recall, remember, update, forget, search, list, stats, recall-log, config, sync, migrate, cleanup, init
- **6 hooks**, automatic context capture (SessionStart, PreToolUse, PostToolUse, UserPromptSubmit, Stop, PreCompact) with zero manual effort
- **FTS5 search**, BM25-ranked full-text search across all memories
- **Cross-editor**, Claude Code (reference adapter) and Cursor (full hook + MCP wiring; some capabilities tracked against upstream Cursor platform work); Codex, Copilot, and Windsurf get a curated rules template
- **Local-first**, SQLite + JSON files; code and memory content never leave your machine. Anonymized event tallies are sent to PostHog only when you opt in via `AIDE_TELEMETRY=on`
